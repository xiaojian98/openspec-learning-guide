# 01 - 为什么需要 OpenSpec

> 理解问题，才能理解解决方案

## 你是否经历过这些场景？

### 场景一：AI 写了一堆代码，结果不是你想要的

```
你：帮我加个登录功能

AI：（疯狂输出 500 行代码）

你：呃...我不是这个意思...我要的是简单的邮箱登录

AI：（再输出 300 行代码）

你：...算了，我自己写吧
```

**问题**：需求理解不一致，浪费时间。

### 场景二：改来改去，永远做不完

```
第一天：加个按钮
第二天：按钮位置不对，改一下
第三天：颜色也不对，再改一下
第四天：其实我想的是另一个按钮...
```

**问题**：没有明确的规范，反复返工。

### 场景三：多人协作，各说各话

```
开发者 A：我认为这个功能是这样的
开发者 B：不，产品说是那样的
产品经理：我说的不是这个意思！
```

**问题**：缺乏统一的"真实状态"文档。

## OpenSpec 的解决方案

### 核心理念：先对齐，再写码

```
┌─────────────────────────────────────────────────────────────────┐
│                     传统方式 vs OpenSpec                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   传统方式：                                                     │
│   ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐        │
│   │  需求  │───▶│ AI写码 │───▶│ 不对！ │───▶│ 重来！ │        │
│   └────────┘    └────────┘    └────────┘    └────────┘        │
│        │                                        │               │
│        └────────────────────────────────────────┘               │
│                      无限循环 💀                                 │
│                                                                 │
│   ─────────────────────────────────────────────────────────────│
│                                                                 │
│   OpenSpec 方式：                                               │
│   ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐        │
│   │  需求  │───▶│ 写规范 │───▶│ 确认？ │───▶│ AI写码 │───▶ ✅  │
│   └────────┘    └────────┘    └────────┘    └────────┘        │
│                    │              │                             │
│                    │              └── 确认无误后再写代码         │
│                    └── 先用文档对齐理解                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 三大核心价值

#### 1. 规范先行（Spec-First）

在写代码之前，先写清楚：
- **要做什么**（What）
- **为什么做**（Why）
- **怎么验证**（How to verify）

#### 2. 增量描述（Delta Specs）

不需要重写整个文档，只描述"变化"的部分：

```markdown
## ADDED Requirements      ← 新增的功能
## MODIFIED Requirements   ← 修改的功能
## REMOVED Requirements    ← 删除的功能
```

#### 3. 流体工作流（Fluid Workflow）

不是死板的"规划→实现→完成"，而是随时可以：
- 回到上一步修改
- 并行处理多个变更
- 跳过不需要的步骤

## OpenSpec 适合你吗？

### ✅ 强烈推荐

| 场景 | 原因 |
|------|------|
| AI 编程新手 | 避免 AI 理解偏差，减少返工 |
| 复杂功能开发 | 先理清思路，再动手 |
| 多人协作项目 | 统一规范，方便沟通 |
| 长期维护项目 | 规范文档就是最好的文档 |

### ⚠️ 可选

| 场景 | 原因 |
|------|------|
| 简单 bug 修复 | 直接改更快的 |
| 一次性脚本 | 不需要长期维护 |
| 个人小项目 | 看个人偏好 |

### ❌ 可能不需要

| 场景 | 原因 |
|------|------|
| 紧急修复 | 时间紧迫，先解决问题 |
| 纯实验性代码 | 不确定会不会保留 |

## 与其他工具对比

### vs. 传统需求文档

| 特性 | 传统文档 | OpenSpec |
|------|---------|----------|
| 与代码的关系 | 分离，容易过时 | 紧密结合，自动同步 |
| 更新方式 | 手动维护 | AI 辅助生成 |
| 粒度 | 粗粒度 | 细粒度（每个变更） |

### vs. Spec Kit（GitHub）

| 特性 | Spec Kit | OpenSpec |
|------|---------|----------|
| 复杂度 | 较重，有阶段门 | 轻量，流体式 |
| AI 集成 | 有限 | 深度集成 |
| 自定义 | 固定结构 | 可自定义 Schema |

### vs. Kiro（AWS）

| 特性 | Kiro | OpenSpec |
|------|------|----------|
| 工具绑定 | 必须用他们的 IDE | 支持任意工具 |
| 模型绑定 | 限 Claude | 支持任意模型 |
| 开源 | 否 | 是（MIT） |

---

## 总结

OpenSpec 解决的核心问题是：

> **让 AI 和人类在写代码之前，先对齐要做什么。**

通过"规范先行"的方式，减少返工，提高效率，让开发更愉快。
