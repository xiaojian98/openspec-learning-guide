# 15 - 多变更并行

> 高效处理多个同时进行的变更

## 为什么需要多变更？

```
┌─────────────────────────────────────────────────────────────────┐
│                    多变更场景                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   场景 1：功能 A 和功能 B 都在做                                 │

│   └── charlie/add-tests   （Charlie 的变更）                    │
```

## 指定变更操作

### 使用斜杠命令

```
# 指定变更名
/opsx:apply add-dark-mode
/opsx:archive fix-login-bug

# 不指定时，AI 会推断或询问
/opsx:apply
> 检测到多个变更：add-dark-mode, fix-login-bug
> 请选择要操作的变更：
```

### 使用 CLI

```bash
# 查看特定变更
openspec show add-dark-mode

# 验证特定变更
openspec validate add-dark-mode

# 归档特定变更
openspec archive add-dark-mode
```

## 变更隔离

### 每个变更有独立的 specs

```
openspec/changes/
├── add-dark-mode/
│   └── specs/
│       └── ui/
│           └── spec.md      # 只描述 dark mode 相关的 delta
│
└── fix-login-bug/
    └── specs/
        └── auth/
            └── spec.md      # 只描述 login bug 相关的 delta
```

### 不会冲突的原因

1. **Delta Specs 只描述变化**：不是完整规范
2. **不同目录隔离**：每个变更有自己的 `specs/` 目录
3. **归档时合并**：只有归档时才合并到主规范

## 最佳实践

### 1. 变更命名规范

```
✅ 好的命名：
   - add-dark-mode
   - fix-login-timeout
   - refactor-auth-module
   - user/alice/add-api

❌ 不好的命名：
   - update（太模糊）
   - fix（太模糊）
   - 我的变更（非英文）
```

### 2. 控制并行数量

```
建议：同时进行 2-3 个变更

原因：
- 太多变更难以管理
- 容易忘记上下文
- 归档时可能冲突
```

### 3. 及时归档完成的变更

```bash
# 查看待归档变更
openspec list

# 归档已完成的
openspec archive add-dark-mode
```

### 4. 使用前缀区分团队

```
openspec/changes/
├── alice/
│   ├── add-api/
│   └── fix-tests/
├── bob/
│   └── update-styles/
└── shared/
    └── refactor-core/
```

## 冲突处理

### 场景：两个变更修改同一 spec

```
add-dark-mode:  MODIFIED auth/spec.md
fix-login-bug:  MODIFIED auth/spec.md
```

**解决方案**：

1. **先归档一个**：让第一个变更先合并
2. **重新生成 delta**：第二个变更的 delta 需要基于新的主规范重新生成
3. **或拆分变更**：如果关联太紧密，考虑合并为一个变更

### 检测潜在冲突

```bash
# 查看变更影响的文件
openspec show add-dark-mode --deltas-only
openspec show fix-login-bug --deltas-only

# 对比是否有重叠
```
