# 04 - 基础工作流

> 掌握 propose → apply → archive 核心流程

## 工作流概览

```
┌─────────────────────────────────────────────────────────────────┐
│                    OpenSpec 基础工作流                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   /opsx:explore        思考模式（可选）                          │
│        │                                                        │
│        │  不确定要做什么时，先探索                                │
│        ▼                                                        │
│   /opsx:propose       创建变更                                  │
│        │                                                        │
│        │  生成 proposal + specs + design + tasks                │
│        ▼                                                        │
│   /opsx:apply         实现任务                                  │
│        │                                                        │
│        │  按 tasks.md 逐步实现                                   │
│        ▼                                                        │
│   /opsx:archive       归档变更                                  │
│        │                                                        │
│        │  合并 deltas，移动到 archive/                           │
│        ▼                                                        │
│      ✅ 完成                                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

## 1. Explore（思考模式）

### 什么时候用？

- 不确定要做什么
- 想先探索想法
- 需要澄清需求

### 使用方式

```
/opsx:explore
```

### 特点

- **不创建任何文件**
- 只是和 AI "对话思考"
- 帮助理清思路

### 示例对话

```
你：/opsx:explore

AI：好，让我们思考一下。你想做什么？

你：我想加个用户登录功能，但不确定用什么方案...

AI：登录功能有几种常见方案：
    1. 邮箱 + 密码（传统）
    2. OAuth（GitHub/Google）
    3. Magic Link（无密码）

    你的用户群体是什么？安全性要求如何？

你：主要是内部系统，用 GitHub OAuth 吧...

AI：好的，GitHub OAuth 是个不错的选择。让我们开始提案？

你：好的

AI：准备好后，运行 /opsx:propose 添加 GitHub OAuth 登录
```

## 2. Propose（创建变更）

### 使用方式

```
/opsx:propose <变更描述>
```

### 示例

```
/opsx:propose 添加 GitHub OAuth 登录
```

### 会发生什么？

1. 创建变更目录 `openspec/changes/add-github-oauth/`
2. 生成以下文件：
   - `proposal.md` - 为什么做、做什么
   - `specs/auth/spec.md` - 具体要求（Delta）
   - `design.md` - 技术方案
   - `tasks.md` - 任务清单

### 查看生成的变更

```bash
# 列出所有变更
openspec list

# 查看变更状态
openspec status

# 查看变更详情
openspec show add-github-oauth
```

### 审阅和修改

在实现之前，你应该审阅生成的文档：

```bash
# 查看 proposal
cat openspec/changes/add-github-oauth/proposal.md

# 查看 tasks
cat openspec/changes/add-github-oauth/tasks.md
```

**如果发现问题，直接编辑这些文件即可。**

## 3. Apply（实现任务）

### 使用方式

```
/opsx:apply
```

或指定变更：

```
/opsx:apply add-github-oauth
```

### 会发生什么？

AI 会按照 `tasks.md` 中的任务列表逐个实现：

```markdown
## 1. 环境准备

- [x] 1.1 安装 passport-github2 依赖
- [x] 1.2 配置 GitHub OAuth App

## 2. 后端实现

- [ ] 2.1 创建 OAuth 路由
- [ ] 2.2 实现回调处理
- [ ] 2.3 添加用户模型
```

### 任务完成时

每个任务完成后，AI 会自动更新任务状态：

```markdown
- [x] 2.1 创建 OAuth 路由
```

### 中途遇到问题

如果实现过程中发现问题：

```
## Implementation Paused

**Change:** add-github-oauth
**Progress:** 5/8 tasks complete

### Issue Encountered
GitHub OAuth 需要企业账号才能使用组织登录

**Options:**
1. 改用个人 GitHub OAuth
2. 升级到企业账号
3. 改用其他登录方式
```

你可以选择一个方案，或者让 AI 更新设计文档。

## 4. Archive（归档变更）

### 使用方式

```
/opsx:archive
```

或指定变更：

```
/opsx:archive add-github-oauth
```

### 会发生什么？

```
归档前：
openspec/
├── specs/
│   └── auth/
│       └── spec.md           ← 当前规范
└── changes/
    └── add-github-oauth/
        ├── proposal.md
        ├── design.md
        ├── tasks.md
        └── specs/
            └── auth/
                └── spec.md   ← Delta 规范

归档后：
openspec/
├── specs/
│   └── auth/
│       └── spec.md           ← 合并后的新规范
└── changes/
    └── archive/
        └── 2025-01-20-add-github-oauth/  ← 归档的变更
            ├── proposal.md
            ├── design.md
            ├── tasks.md
            └── specs/
                └── auth/
                    └── spec.md
```

### 归档选项

| 选项 | 说明 |
|------|------|
| `--skip-specs` | 跳过 spec 合并（用于非功能变更） |
| `--no-validate` | 跳过验证（不推荐） |
| `-y, --yes` | 跳过确认提示 |

### 示例

```bash
# 跳过 spec 合并（比如只是改了 CI 配置）
openspec archive --skip-specs

# 强制跳过确认
openspec archive -y
```

## 常用命令速查

### CLI 命令

| 命令 | 说明 |
|------|------|
| `openspec init` | 初始化项目 |
| `openspec list` | 列出变更 |
| `openspec status` | 查看变更状态 |
| `openspec show <name>` | 查看变更详情 |
| `openspec validate` | 验证变更 |
| `openspec archive <name>` | 归档变更 |
| `openspec update` | 更新 AI 指令文件 |

### 斜杠命令

| 命令 | 说明 |
|------|------|
| `/opsx:explore` | 思考模式 |
| `/opsx:propose` | 创建变更 |
| `/opsx:apply` | 实现任务 |
| `/opsx:archive` | 归档变更 |

## 完整示例

### 场景：添加深色模式

```
你：/opsx:explore
    我想给网站加个深色模式，有什么要注意的吗？

AI：深色模式有几个关键点：
    1. CSS 变量 vs CSS-in-JS
    2. 系统偏好检测
    3. 用户选择持久化

    你的技术栈是什么？

你：React + Tailwind

AI：Tailwind 有内置的 dark: 前缀，很方便。建议：
    - 使用 localStorage 持久化
    - 默认跟随系统
    - 提供手动切换开关

    准备好开始了吗？

你：/opsx:propose 添加深色模式支持

AI：创建了变更 add-dark-mode/
    ✓ proposal.md
    ✓ specs/ui/spec.md
    ✓ design.md
    ✓ tasks.md

    查看生成的文件后，运行 /opsx:apply 开始实现

你：/opsx:apply

AI：开始实现任务...
    ✓ 1.1 添加 CSS 变量
    ✓ 1.2 创建 ThemeProvider
    ✓ 1.3 添加切换按钮
    ✓ 1.4 实现持久化

    所有任务完成！

你：/opsx:archive

AI：归档完成！
    变更已移动到 openspec/changes/archive/2025-01-20-add-dark-mode/
```
