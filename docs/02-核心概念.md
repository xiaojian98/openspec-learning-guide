# 02 - 核心概念

> 理解 OpenSpec 的核心概念是高效使用的基础

## 1. Specs（规范）

### 是什么？

Specs 是**系统当前行为的真实状态**——描述你的系统"现在"是怎么工作的。

### 存放在哪里？

```
openspec/
└── specs/
    ├── auth/
    │   └── spec.md      # 认证相关规范
    ├── user/
    │   └── spec.md      # 用户相关规范
    └── api/
        └── spec.md      # API 相关规范
```

### 包含什么？

每个 spec.md 包含：

```markdown
# Auth Specification

## Purpose
描述系统的认证机制。

## Requirements

### Requirement: 用户登录
系统必须验证用户身份，成功后颁发会话凭证。

#### Scenario: 邮箱登录成功
- GIVEN 用户有有效账户
- WHEN 用户输入正确的邮箱和密码
- THEN 系统颁发 JWT Token
- AND 用户被重定向到首页

#### Scenario: 密码错误
- GIVEN 用户有有效账户
- WHEN 用户输入错误的密码
- THEN 系统显示"密码错误"
- AND 不颁发 Token
```

### 关键点

| 元素 | 说明 |
|------|------|
| `## Purpose` | 这个规范描述什么 |
| `### Requirement:` | 一个具体的行为要求 |
| `#### Scenario:` | 一个可验证的场景 |
| SHALL/MUST | 必须遵守 |
| SHOULD | 建议遵守 |

## 2. Changes（变更）

### 是什么？

Change 是**一个提议的修改**——你想对系统做什么改动。

### 存放在哪里？

```
openspec/
└── changes/
    ├── add-login/           # 进行中的变更
    │   ├── proposal.md      # 为什么做、做什么
    │   ├── design.md        # 技术方案
    │   ├── tasks.md         # 任务清单
    │   └── specs/           # 变更的规范（Delta）
    │       └── auth/
    │           └── spec.md
    └── archive/             # 已完成的变更
        └── 2025-01-15-add-login/
```

### 为什么用文件夹？

1. **独立隔离**：每个变更互不干扰
2. **完整上下文**：proposal + design + tasks + specs 都在一起
3. **方便审查**：打开一个文件夹就能看懂整个变更
4. **历史追溯**：归档后保留完整记录

---

## 3. Deltas（增量规范）

### 是什么？

Delta 是**只描述变化部分**的规范，而不是重写整个规范。

### 为什么用 Delta？

```
传统方式（全量）：
┌────────────────────────────────────────────┐
│ # Auth Specification                       │
│                                            │
│ ## Requirements                            │
│                                            │
│ ### Requirement: 用户登录                   │
│ （完整内容...）                             │
│                                            │
│ ### Requirement: 密码重置  ← 只改这里       │
│ （完整内容...）                             │
│                                            │
│ ### Requirement: 会话管理                   │
│ （完整内容...）                             │
│                                            │
│ ...还有 20 个其他需求...                    │
└────────────────────────────────────────────┘
需要重写整个文件！

Delta 方式（增量）：
┌────────────────────────────────────────────┐
│ # Delta for Auth                           │
│                                            │
│ ## MODIFIED Requirements                   │
│                                            │
│ ### Requirement: 密码重置                   │
│ 系统必须支持手机号重置密码。               │
│ （之前：只支持邮箱）                        │
│                                            │
└────────────────────────────────────────────┘
只写变化的部分！
```

### Delta 操作

| 操作 | 说明 | 归档时 |
|------|------|--------|
| `## ADDED Requirements` | 新增功能 | 追加到主规范 |
| `## MODIFIED Requirements` | 修改功能 | 替换主规范中的对应项 |
| `## REMOVED Requirements` | 删除功能 | 从主规范中移除 |

## 4. Artifacts（产物）

### 是什么？

Artifact 是变更中的**各种文档**，每个有特定用途。

### 有哪些 Artifacts？

```
┌─────────────────────────────────────────────────────────────────┐
│                      Artifacts 流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   proposal ──────────► specs ──────────► design ──────────►    │
│       │                   │                   │                 │
│     为什么              具体要求            技术方案            │
│     做什么              什么变化            怎么实现            │
│                                                                 │
│                              │                                  │
│                              ▼                                  │
│                           tasks                                 │
│                              │                                  │
│                          任务清单                               │
│                          分几步做                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| Artifact | 用途 | 关键内容 |
|----------|------|---------|
| `proposal.md` | 为什么做、做什么 | Intent、Scope、Impact |
| `specs/*.md` | 具体要求 | ADDED/MODIFIED/REMOVED |
| `design.md` | 技术方案 | 架构决策、数据流、文件变更 |
| `tasks.md` | 任务清单 | 可勾选的任务列表 |

---

## 5. Schemas（工作流模板）

### 是什么？

Schema 定义了**有哪些 Artifacts 以及它们的依赖关系**。

### 默认 Schema（spec-driven）

```
         proposal
         (根节点)
            │
     ┌──────┴──────┐
     ▼             ▼
   specs        design
     │             │
     └──────┬──────┘
            ▼
          tasks
```

- `proposal` 没有依赖，可以先创建
- `specs` 和 `design` 依赖 `proposal`
- `tasks` 依赖 `specs` 和 `design`

### 自定义 Schema

你可以创建自己的 Schema，比如：

```
research → proposal → tasks
```

或者：

```
proposal → security-review → specs → design → tasks
```
