# 05 - 项目配置详解

> 深入理解 config.yaml 配置

## 配置文件位置

```
your-project/
└── openspec/
    └── config.yaml    ← 项目配置文件
```

---

## 完整配置示例

```yaml
# openspec/config.yaml

# 默认使用的 Schema
schema: spec-driven

# 项目背景（会注入到所有 AI 指令）
context: |
  语言：中文（简体）
  所有产出物必须用简体中文撰写。

  技术栈: TypeScript + React 18 + Node.js 20
  数据库: PostgreSQL + Prisma ORM
  API 风格: RESTful, JSON 响应
  测试框架: Vitest + Playwright
  代码风格: ESLint + Prettier, 严格 TypeScript

  我们重视：
  - 向后兼容性
  - 性能优化
  - 安全性

# 按 Artifact 类型分类的规则
rules:
  proposal:
    - 必须包含回滚方案
    - 必须标识受影响的团队
    - 必须评估性能影响
  specs:
    - 使用 Given/When/Then 格式写场景
    - 引用现有模式，不要重复发明轮子
  design:
    - 复杂流程必须有流程图
    - 数据库变更必须有迁移计划
  tasks:
    - 每个任务应该能在 1 小时内完成
```

---

## 配置字段详解

### schema

默认使用的工作流 Schema。

```yaml
schema: spec-driven
```

**可用值：**
- `spec-driven`（默认）
- 自定义 Schema 名称

**覆盖优先级：**
1. CLI 参数：`--schema <name>`
2. 变更元数据：`.openspec.yaml`
3. 项目配置：`config.yaml`
4. 默认值：`spec-driven`

---

### context

项目背景信息，会注入到所有 AI 指令中。

```yaml
context: |
  技术栈: TypeScript + React
  测试: Vitest
```

**注入格式：**

```xml
<context>
技术栈: TypeScript + React
测试: Vitest
</context>

<template>
[Schema 的模板内容]
</template>
```

**最佳实践：**
- 包含技术栈信息
- 说明团队约定
- 列出重要约束
- 使用中文描述

**大小限制：** 50KB

---

### rules

按 Artifact 类型分类的规则，只在创建对应 Artifact 时注入。

```yaml
rules:
  proposal:
    - 规则 1
    - 规则 2
  specs:
    - 规则 1
  design:
    - 规则 1
  tasks:
    - 规则 1
```

**注入格式（以 specs 为例）：**

```xml
<context>
[context 内容]
</context>

<rules>
- 规则 1
- 规则 2
</rules>

<template>
[specs 的模板内容]
</template>
```

**可用的 Artifact ID：**

| Schema | Artifact IDs |
|--------|-------------|
| spec-driven | proposal, specs, design, tasks |
| rapid | proposal, tasks |
| 自定义 | 自定义的 artifact id |

## 常见配置场景

### 场景一：纯中文项目

```yaml
schema: spec-driven

context: |
  语言：中文（简体）
  所有产出物必须用简体中文撰写。

  技术栈: Vue 3 + TypeScript + Vite
```

### 场景二：企业级项目

```yaml
schema: enterprise

context: |
  公司：XX 科技有限公司
  项目：内部管理系统

  技术栈: Java 17 + Spring Boot + PostgreSQL
  部署: Kubernetes + AWS

  安全要求：
  - 所有 API 必须有认证
  - 敏感数据必须加密
  - 操作必须有审计日志

rules:
  proposal:
    - 必须包含安全影响评估
    - 必须包含合规性检查
  design:
    - 必须包含部署方案
    - 必须包含回滚方案
```

### 场景三：快速迭代项目

```yaml
schema: rapid

context: |
  项目类型：原型验证
  迭代周期：1 周

  技术栈: Next.js + TypeScript

  优先级：速度 > 完美
```

## 验证配置

### 检查配置是否生效

```bash
# 查看某个 artifact 的指令（应该包含你的 context）
openspec instructions proposal --change my-change

# JSON 格式输出
openspec instructions proposal --change my-change --json
```

### 常见问题

#### Q: 配置没有生效？

检查：
1. 文件位置是否正确：`openspec/config.yaml`
2. YAML 语法是否正确
3. 字段名是否正确

#### Q: rules 中的 Artifact ID 提示未知？

```bash
# 查看当前 Schema 支持的 Artifact ID
openspec schemas --json
```

## 高级配置

### 使用环境变量

```yaml
context: |
  环境：${NODE_ENV}
  API 地址：${API_BASE_URL}
```

### 多行规则

```yaml
rules:
  proposal:
    - |
      回滚方案格式：
      1. 回滚触发条件
      2. 回滚步骤
      3. 数据恢复方案
```
